## Overview

**Page Purpose:** Complete the entire purchase in one continuous panel — choose delivery, fill in contact/address, pay, book mounting, see confirmation.

**Architecture:** The checkout is a 4th panel in the FlowShell system. It slides up from below over the results view (which stays rendered underneath). No page navigation occurs — only React state transitions between steps. The URL does NOT change.

**User Situation:** Harriet tapped "Velg disse" on a product card. The tire was added to cart (if not already there), then the checkout panel slid up. She sees the delivery options immediately.

**Success Criteria:**
- She picks Fjellhamar and fills in her info without leaving the flow
- Payment completes without any page redirect
- Booking time appears and she picks a slot
- Confirmation screen scrolls in — she knows it's done

**Entry Points:**
- Tap "Velg disse" on 01.2-Product Cards or 01.3-Product Detail → adds to cart → panel slides up
- Tap buy button when tire already in cart → panel slides up (no duplicate added)

**Exit Points (within panel):**
- Back button / scroll-up gesture → navigate back through steps (or close panel → back to results)
- Order placed → scroll down to inline confirmation (back nav locked from there)

## Architecture

### FlowShell Panel Stack

```
z: 50 — Fixed service buttons (language, cart badge) — always visible
z: 30+— CHECKOUT PANEL ← active when view === "checkout"
z: 20 — QUANTITY-SHOP PANEL
z: 10 — RESULTS PANEL ← pushed off-top (-translate-y-full) when checkout is open
z: 0  — HOME PANEL
```

All panels use `absolute inset-0`, `overflow-y-auto`, `transition-transform duration-500`.

Checkout panel: `translate-y-0` when active, `translate-y-full` when inactive.

### Step State Machine

Steps are managed with React state (`step: string`) inside `CheckoutPanelContent`. No URL params involved.

```
"delivery" → "address" → "payment" → "booking" (workshop only) → [order placed]
```

Back navigation:
```
"delivery" → onBack() → setView("results")
"address"  → setStep("delivery")
"payment"  → setStep("address")
"booking"  → setStep("payment")
```

### Back Gesture

- **Scroll-up at top of panel** (wheel deltaY < −40): triggers back navigation with 600ms lockout to prevent double-fire
- **Touch swipe down** (> 60px from top): same

## Loading State

When the checkout panel opens, it fetches cart, customer, shipping methods, and payment methods in parallel. While loading, a **skeleton** is shown that mirrors the real layout:

- Three bordered radio rows (delivery options skeleton)
- A grid of input field placeholders (address form skeleton)
- A dimmed payment section heading

No spinner. The skeleton has `animate-pulse` shimmer.

## Sub-step: 01.4.1 — Leveringsmåte

**Component:** `Shipping`

### Display Order

Workshops are shown **first**, home delivery **last**:
1. Drammen — montering inkl. (pickup)
2. Fjellhamar — montering inkl. (pickup)
3. Hjemlevering (home delivery, calculated price)

Detection: options with `type.code.startsWith("pickup-")` or name containing "montering" are treated as workshop/pickup.

### Layout

```
Leveringsmåte                          ✓ (when completed)
┌─────────────────────────────────────┐
│ ● Drammen — montering inkl.   0 kr  │  ← selected
└─────────────────────────────────────┘
┌─────────────────────────────────────┐
│ ○ Fjellhamar — montering inkl. 0 kr │
└─────────────────────────────────────┘
┌─────────────────────────────────────┐
│ ○ Hjemlevering               699 kr │
└─────────────────────────────────────┘

[ Fortsett ]
──────────────────────────────────────
```

- **"Fortsett" enabled:** when `shippingMethodId` state is set (immediately on radio select — does not wait for cart refresh)
- **"Endre" link:** visible in collapsed state when step is complete
- **Collapsed state:** only the heading is shown (no detail row)

## Sub-step: 01.4.2 — Kundeopplysninger / Leveringsadresse

**Component:** `Addresses`

### Adaptive heading

The heading and form fields adapt to the chosen delivery method:

| Delivery method | Heading | Fields shown |
|-----------------|---------|-------------|
| Workshop (montering) | **Kundeopplysninger** | Fornavn, Etternavn, E-post, Telefon, Bilregistreringsnummer |
| Home delivery | **Leveringsadresse** | Full address form + billing address toggle |

Detection: `cart.shipping_methods[0].name.toLowerCase().includes("montering")`

### Form submission

Uses `setAddressesInPanel` server action (no redirect). On success, calls `onStepChange("payment")`.

### Collapsed state

Only heading shown. Checkmark appears when `cart.email` is populated.

---

## Sub-step: 01.4.3 — Betaling

**Component:** `Payment` wrapped in `PaymentWrapper` (Stripe Elements)

### Workshop orders

Shows **"Continue to booking"** button instead of PaymentButton directly. On click → `setStep("booking")`.

### Home delivery orders

Shows Stripe `PaymentButton` directly. On payment confirmed → calls `onSuccess(orderId)` → triggers inline confirmation.

### Collapsed state

Shows payment method name + card brand icon.

---

## Sub-step: 01.4.4 — Booking (workshop orders only)

**Component:** `Booking`

Conditionally rendered — only when `cart.shipping_methods[0].name.includes("montering")`. For home-delivery carts, `Booking` renders `null`.

### Flow

1. Date picker — next 5 weekdays (excludes weekends)
2. Time picker — 6 slots: 08:00, 09:30, 11:00, 13:00, 14:30, 16:00
3. On time selected → `saveBookingToCart(date, time, workshopName)` saves to `cart.metadata`
4. PaymentButton enabled when date + time selected
5. On payment confirmed → `onSuccess(orderId)` → inline confirmation

### Data persistence

Booking is saved to `cart.metadata`:
```json
{
  "booking_date": "2026-04-10",
  "booking_time": "10:00",
  "booking_workshop": "Fjellhamar"
}
```

This persists through order placement. `order.metadata` contains the same fields and is displayed on the order confirmed page.

## Sub-step: 01.4.5 — Bekreftelse (inline confirmation)

**Component:** `OrderConfirmedInline` (inside CheckoutPanelContent)

Triggered when `onSuccess(orderId)` is called from Payment or Booking components.

### Behavior

1. `orderId` state is set
2. Checkout steps are hidden (`display: none`)
3. Confirmation div scrolls into view (`scrollIntoView({ behavior: "smooth" })`)
4. Back gesture is **locked** — `backLocked.current = true`, `orderId` check prevents re-attachment of gesture handlers

### Layout

```
+──────────────────────────+
│                          │
│      ✓ (green circle)    │
│                          │
│   Takk for bestillingen! │
│   Ordrenummer: XXXXXXXX  │
│   Du vil motta e-post... │
│                          │
│  ┌────────────────────┐  │  (workshop only, when booking was made)
│  │ Monteringstidspunkt│  │
│  │ Fjellhamar         │  │
│  │ 2026-04-10, 10:00  │  │
│  └────────────────────┘  │
│                          │
│  [ Tilbake til forsiden ] │

```

## Technical Notes

- **No page navigation:** The panel uses `setView("checkout")` in FlowShell, not `router.push`. URL remains unchanged.
- **Cart idempotency:** Before calling `addToCart`, FlowShell checks `cart.items` for an existing line with the same variant ID. If found, skips add.
- **Fresh cart on open:** `CheckoutPanelContent` is keyed with `checkoutKey` — incremented each time checkout opens — ensuring a fresh mount and fresh data fetch each time.
- **Stripe Elements:** `PaymentWrapper` initializes Stripe with `NEXT_PUBLIC_STRIPE_KEY`. Stripe Elements are client-side only.
- **Server actions:** `setAddressesInPanel`, `placeOrderInPanel`, `saveBookingToCart` are server actions that return values instead of calling `redirect()`.
- **Scroll-up lockout:** Back gesture uses a 600ms lockout ref (`backLocked`) to prevent double-fire on trackpad momentum scroll.
