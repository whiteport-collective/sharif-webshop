# 01.5 — Payment

## Page Metadata

| Property | Value |
|----------|-------|
| **Scenario** | 01: Harriet's Tire Purchase |
| **Page Number** | 01.5 |
| **Platform** | Mobile (iPhone, portrait) |

## Overview

**Page Purpose:** Harriet pays for her tires. This is the clean hand-off to Klarna/Qliro. The guided flow ends here — the payment page is the external checkout system.

**Entry Context:** She tapped "Betal nå — 1 996 kr" on the quantity/shop view. A new page loads with the payment provider's checkout.

**Exit Action:** Completes payment → redirected back to Sharif.no booking view (01.6)

## On-Page Interactions

This view is controlled by Klarna/Qliro — not custom-designed. Sharif passes:
- Order details (4x Powertrac Adamas H/P 185/65R15)
- Total amount (1 996 kr)
- Customer return URL (→ booking view)

The payment provider handles:
- Customer identification (BankID, email, phone)
- Payment method selection (card, Vipps, invoice, installments)
- Private/company toggle
- Address collection

## Content Elements

| Element | Type | Content |
|---------|------|---------|
| Order summary | Passed data | Product, quantity, total — displayed by payment provider |
| Payment methods | Klarna/Qliro | Card, Vipps, invoice, installments |
| Customer details | Klarna/Qliro | Name, address, phone — handled by provider |
| Private/Company | Klarna/Qliro | Toggle — company checkout handled natively |

## Design Notes

- This is NOT a custom-designed view. It's the payment provider's embedded checkout.
- The page reload from the guided flow to the checkout is the "clean cut" — architecturally, this is where Sharif's frontend ends and the e-commerce platform begins.
- For the POC/demo, this can be a Klarna sandbox or a simulated checkout.
- The return URL after successful payment should include order reference data.
- The visual break is acceptable here — Harriet has already made all her decisions. She's just paying.
