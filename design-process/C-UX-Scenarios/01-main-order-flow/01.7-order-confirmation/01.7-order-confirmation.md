## Overview

**Page Purpose:** Close the loop on the purchase. After the order is placed, the checkout stays in a two-column layout: all four completed steps collapse on the left showing the customer's filled-in data with green checkmarks, followed by the celebration message, rating prompt, and AI support chat. The cart sidebar stays visible and sticky on the right.

**User Situation:** Harriet has completed payment and (optionally) booked a mounting time. The checkout steps collapse to show compact summaries of what she entered (no labels — just values). The confirmation section scrolls into view below the steps. No back navigation — the order is placed.

**Success Criteria:**
- Harriet sees her complete order at a glance in both columns
- Completed steps give visual satisfaction with ✅ checkmarks
- Her vehicle's registration plate is shown in the customer details — tying the order to her car
- She can rate her experience (optional)
- She can ask questions via AI chat without calling support

**Entry Points:**
- Order placed successfully from CheckoutPanelContent (`onSuccess` callback)

**Exit Points:**
- "Tilbake til forsiden" button → home
- Menu navigation → any page
- No back navigation — this is the terminal state

## AI Behavior

The AI is Claude Haiku acting as a Sharif customer service agent.

**System prompt context passed on each request:**
- Order ID, product name, dimension, quantity, price
- Booked mounting time and shop
- Customer email
- Vehicle registration number

**Capable of handling:**
- Wrong tire dimension ordered → offer to initiate exchange
- Cancellation requests → explain process, offer alternatives
- Mounting time questions → confirm details, link to rebooking
- General product questions → answer from product data
- Anything else → warm handoff to human support (phone number)

**Tone:** Friendly, concise. Norwegian by default, switches to English if user writes in English.

**Data persistence:** All messages stored in `dialog_messages` linked to `dialog.id` → `order_id`. Accessible to support agents via admin view.

## Dialog Data Model

```
dialog
├── id
├── cart_id          (set when cart created)
├── order_id         (set when order placed)
├── customer_id      (if logged in)
├── customer_email
├── product_handle
├── dimension
├── registration_nr   (vehicle registration from search)
├── outcome          purchased | abandoned | support
├── rating           (1-5, set when stars tapped)
├── ai_summary       (generated on abandon)
├── created_at
├── converted_at
└── abandoned_at

dialog_messages
├── id
├── dialog_id
├── role             user | assistant | agent
├── content
└── created_at
```

## Storyboard: AI Chat Flow

The AI chat starts as an empty textarea. When the customer sends a message, chat bubbles appear inline above the textarea, pushing the order confirmation upward and out of view. The conversation becomes the focus.

```

+─────────────────────────────────+

FRAME 2: Customer types and sends a message
+─────────────────────────────────+
| Snakk med var AI                |
|                                 |
|      ┌──────────────────────┐   |
|      │ Kan jeg endre til    │   |  ← User bubble (dark, right-aligned)

| +─────────────────────────────+ |
| | +                       [>] | |

+─────────────────────────────────+

FRAME 3: AI responds — order confirmation scrolled out of view
+─────────────────────────────────+
|      ┌──────────────────────┐   |
|      │ Kan jeg endre til    │   |  ← User message
|      │ bredere dekk?        │   |
|      └──────────────────────┘   |
| ┌───────────────────────────┐   |
| │ Ja, 225/45R17 passer til  │   |  ← AI bubble (light, left-aligned)
| │ EK 12345! Jeg kan starte  │   |
| │ en bytteordre.            │   |
| └───────────────────────────┘   |
|                                 |
|      ┌──────────────────────┐   |
|      │ Ja takk, kjor pa!    │   |  ← User reply
|      └──────────────────────┘   |
|                                 |
| +─────────────────────────────+ |
| |                             | |  ← Textarea stays at bottom
| +─────────────────────────────+ |
| | +                       [>] | |
| +─────────────────────────────+ |
+─────────────────────────────────+
```

| Frame | Trigger | What happens |
|-------|---------|--------------|
| 1 → 2 | Customer taps send | Message appears as dark bubble (right-aligned). Textarea clears. Send button shows spinner briefly. Celebration header scrolls up. |
| 2 → 3 | AI responds (~1-2s) | AI bubble appears (light, left-aligned) below user message. New messages push older content up. Textarea stays pinned at bottom of viewport. |
| Ongoing | More messages | Chat grows upward. The order confirmation, steps, and celebration are still there — just scrolled above the fold. Customer can scroll up to see them. |

**Key behavior:** The textarea is always visible at the bottom. Messages grow upward. The order confirmation is never removed — it's just pushed up by the conversation. This means the page gets longer, not that content is replaced.

## Design Decisions

| Decision | Rationale |
|----------|-----------|
| ✅ checkmarks on completed steps | Gives visual satisfaction and sense of accomplishment |
| No labels on step data | Compact, scannable — the step title provides context |
| Registration plate in customer details | Ties the order to the customer's vehicle — the whole reason they're here |
| No suggestion chips in AI chat | Don't plant doubt or ideas — if the customer has no issues, let them leave happy |
| Two-column layout with sticky cart | Customer always sees what they ordered while reviewing confirmation |
| No address field | Workshop orders don't need delivery address — just contact info |

## Technical Notes

- The confirmation is **inline in CheckoutPanelContent** — not a separate FlowShell view
- When `orderId` is set, completed steps show collapsed summaries with ✅ in left column
- `onConfirmationReached()` callback hides the back button in FlowShell header
- Cart sidebar remains visible (sticky) throughout — same component as during checkout
- Registration plate data comes from `cart.metadata.registration_number` (set during tire search)
- Rating stored as `dialog.rating` (integer 1-5)
- AI dialog uses same `dialog` + `dialog_messages` tables as pre-purchase chat
